**CORE_RISKS**
- Wallet funds can be minted without payment. Authenticated users can call `/api/payments/deposits` and credit their own balance immediately, bypassing Stripe/webhooks entirely. [api.py](/home/yinch/Projects/CaMeL_Community_Web/backend/apps/payments/api.py#L144), [services.py](/home/yinch/Projects/CaMeL_Community_Web/backend/apps/payments/services.py#L172)
- Auth policy is internally inconsistent. Registration issues JWTs before email verification, login never checks verification, refresh ignores `is_active`, and bearer auth only checks `user_id`; a banned user with an existing token can keep using the API until token expiry and can likely keep refreshing. Logout is also broken: `refresh.blacklist()` is called without `token_blacklist` installed, and I verified `RefreshToken.blacklist` is absent in this setup. [api.py](/home/yinch/Projects/CaMeL_Community_Web/backend/apps/accounts/api.py#L108), [api.py](/home/yinch/Projects/CaMeL_Community_Web/backend/apps/accounts/api.py#L173), [api.py](/home/yinch/Projects/CaMeL_Community_Web/backend/apps/accounts/api.py#L191), [api.py](/home/yinch/Projects/CaMeL_Community_Web/backend/apps/accounts/api.py#L205), [permissions.py](/home/yinch/Projects/CaMeL_Community_Web/backend/common/permissions.py#L8), [admin_api.py](/home/yinch/Projects/CaMeL_Community_Web/backend/apps/accounts/admin_api.py#L297), [base.py](/home/yinch/Projects/CaMeL_Community_Web/backend/config/settings/base.py#L9)
- Bounty detail data is public when it should almost certainly be role-scoped. Anonymous callers can see all applications, proposals, deliverables, comments, arbitrators, and arbitration votes. [api.py](/home/yinch/Projects/CaMeL_Community_Web/backend/apps/bounties/api.py#L126), [api.py](/home/yinch/Projects/CaMeL_Community_Web/backend/apps/bounties/api.py#L185)
- Bounty read endpoints have side effects. Listing/viewing bounties runs `process_automations()`, which can reopen jobs, deduct hunter credit, and auto-approve payouts after 7 days. Public detail/search/version endpoints return `system_prompt`, `user_prompt_template`, examples, and version history; the UI exposes the prompt body to any visitor. **MISSING_REQUIREMENTS**
- The draft does not define the actor matrix. At minimum: anonymous user, registered-unverified user, verified user, banned user, inviter/invitee, skill creator/buyer, bounty creator/hunter/arbitrator/admin.
- It omits negative-path coverage: invalid invite, expired reset link, OAuth misconfiguration, insufficient wallet balance, Stripe webhook replay, Celery down, Meilisearch down, SMTP failure, bounty deadline expiry, dispute appeal after settlement.
- It omits privacy/access requirements. The code currently makes some high-risk data public; the draft should explicitly say what is public vs owner-only vs moderator-only for prompts, bounty proposals, deliverables, dispute evidence, and user identity.
- It omits cross-module dependency checks. Registration credit affects bounty eligibility; wallet funding affects skills, tips, and bounties; invite rewards depend on webhook plus Celery; search/recommendations depend on Redis/Meilisearch.
- It omits environment-specific acceptance. WSL local without real SMTP/OAuth/Stripe is materially different from Docker production behind nginx with webhooks, worker, and beat.

**ALTERNATIVE_DIRECTIONS**
- Split the effort into three passes: `security/blockers` first, `flow correctness` second, `deployment/runtime` third. Do not start with ad hoc UI clicking.
- Treat money movement as a hard boundary. Remove or admin-gate direct balance crediting, collapse all tipping/deposit logic into one service, and add idempotency around every settlement path.
- Make all GET endpoints side-effect free. Move bounty automations and recommendation refreshes to Celery beat.
- Define audience-specific serializers. Public, owner, and moderator/admin should not share the same bounty or skill DTOs.
- If skill execution is not ready, disable paid calling explicitly instead of charging users for placeholder output.
- If real-time notifications matter, move SSE to ASGI/WebSocket infra or revert to client polling.

**CANDIDATE_CRITERIA**
- Auth: registration, login, refresh, logout, password reset, email verification, ban/unban, and OAuth all behave according to one explicit policy; banned users lose access immediately; unverified-user policy is enforced consistently.
- Payments: no public path can create balance without confirmed payment; Stripe webhook is idempotent; deposits, tips, skill purchases, bounty escrow, refund, payout, and invite rewards reconcile correctly in balances and transaction logs.
- Skills: only intended public fields are exposed; paid skill calls return real results or are disabled; insufficient-balance and unpublished-skill flows fail cleanly.
- Bounties: visibility rules are enforced by role; GET requests do not mutate data; creator/hunter/admin dispute and settlement flows are idempotent; arbitration cannot double-pay.
- Workshop: publish, vote, comment, tip, and leaderboard flows use correct auth and consistent wallet rules.
- Ops: local WSL and Docker production both boot with web, worker, beat, Postgres, Redis, Meilisearch, static/media, and webhook routes; deployment has a real readiness check; build/type-check/manage-check pass.
